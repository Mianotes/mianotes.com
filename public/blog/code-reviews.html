Running Repeatable Code Reviews with Codex and Mianotes

Mianotes
4 Jun 2026

If you have agents writing code, you need agents reviewing and testing it too. That changes your role. You are still building software, but you are also deciding how the work gets written, checked, documented, and reused.

Mianotes gives every project its own local knowledge hub, starting from a normal folder on your machine. You or your team can collect specs, prompts, documents, manuals, research, code review notes, and LLM output in one place.

Your agents can use that same context through the API and MCP server, so they do not have to rely on incomplete prompts or disconnected files. They work from the knowledge your project has already collected.

I'll show one practical developer workflow.

I'm going to install Mianotes on my MacBook and use it to manage prompts for a GitHub repo. Then I'll connect Codex to Mianotes through MCP, ask Codex to read one of my saved prompts, run a code review, and save the result back into Mianotes.

The repo I'll use is:

Desktop/projects/senseibox

Inside that repo, I create two folders:

Desktop/projects/senseibox/prompts
Desktop/projects/senseibox/qa

The first folder stores reusable prompts. The second stores QA outputs, reviews, findings, and agent reports.

Install Mianotes

Mianotes supports package installs for macOS and Ubuntu.

For most users, this is the right path because it installs Mianotes into the normal system locations and starts the services through the operating system.

macOS

Download the latest macOS package:

https://github.com/Mianotes/install/releases/latest/download/mianotes.pkg

Open the package and follow the installer. Once installed, open http://localhost:8201 in your browser.

You can also manage Mianotes from your Terminal:

mianotes open
mianotes doctor
mianotes status
mianotes logs
mianotes start
mianotes stop
mianotes uninstall

The macOS package includes the runtime tools Mianotes needs, including Python, ripgrep, ffmpeg, and ffprobe. On Apple Silicon, it also includes Tesseract OCR and English OCR data.

First run

Open Mianotes in your browser and create the first user. This first user becomes the admin user. After that, you can start using the web app straight away, but for this workflow we'll create workspaces inside an existing GitHub repo.

Prepare the repo folders

On my MacBook, I'll create two folders inside my Senseibox repo:

mkdir -p ~/Desktop/projects/senseibox/prompts
mkdir -p ~/Desktop/projects/senseibox/qa

The prompts workspace stores reusable prompts as Markdown files. The qa workspace stores the output from agent work, such as code reviews, audits, bug reports, test plans, and release checks.

Add the workspaces in Mianotes

In Mianotes, go to Settings and add a new workspace.

Use this folder path:

~/Desktop/projects/senseibox/prompts

Name it:

Prompts

Then add another workspace.

Use this folder path:

~/Desktop/projects/senseibox/qa

Name it:

QA

Mianotes keeps the notes as Markdown files inside each workspace folder, so your prompt content stays easy to inspect, back up, and commit when you want to.

Create your first prompt note

First, click the home icon in the left navigation bar to return to the workspace list. You should see both workspaces:

Prompts
QA

Click the Prompts workspace to open it. Once inside this workspace:

  1. Create a folder called Code review.
  2. Inside that folder, create a text note with this title: Python performance.
  3. Add this text:
# Prompt

Review this codebase as a senior backend engineer with a performance and production-readiness focus. 

Audit the working tree read-only. Do not modify files.

Look specifically for:

- Full-table or full-workspace scans on request paths
- N+1 queries and per-result follow-up queries
- Eager loading that multiplies rows across multiple child collections
- Python-side aggregation that should be done in SQL
- Missing pagination, limits, timeouts, or streaming
- Large in-memory I/O, including uploads, downloads, URL fetches, ZIP generation, and file parsing
- Missing or weak composite indexes for common filters and sorts
- Filesystem/database mismatches that create slow authorization checks
- Security issues that overlap with performance, especially SSRF, unbounded remote fetches, and public file access checks
- Maintainability issues that make performance fixes risky or hard to test

For each issue, include:

- Severity: Critical / High / Medium / Low
- File and line number or section
- Current behavior
- Why it matters in production
- Specific fix, preferably with a short code sketch when useful
- Expected impact

Then produce a "Best 20/80 Order" section with the safest implementation order.

Be harsh but practical. Prefer fixes that preserve the existing API behavior and response schemas. Call out where a suggested index or query change should be validated with `EXPLAIN QUERY PLAN` before committing to it.

When you save the note, Mianotes writes it as a Markdown file:

~/Desktop/projects/senseibox/prompts/markdown/code-review/python-performance-3ac696c0.md

This is one of the best parts of Mianotes. You're using a web app, but the important content is still plain Markdown, versioned and on disk.

Create an API key for Codex

Next, connect Codex to Mianotes through MCP.

In Mianotes, go to:

Settings > Create an API key

Mianotes saves the key into its service environment file, so the MCP server can use it without passing the token directly to Codex or copying it to your clipboard.

Add the Mianotes MCP server to Codex

First, make sure the MCP command is available:

command -v mianotes-mcp

Now install Codex CLI if you haven't already, then add the Mianotes MCP server:

codex mcp add mianotes --env MIANOTES_CLIENT_NAME=Codex -- mianotes-mcp

Check that Codex can see it:

codex mcp list

You can also check the desktop app: Codex > Settings > MCP servers. Mianotes should appear in the MCP server list.

If Codex was already open, start a new session so it picks up the updated MCP configuration and API key.

Ask Codex to use the prompt

Now you can ask Codex to read the prompt from Mianotes, use it against your codebase, and save the output back into the QA workspace.

Use a prompt like this:

Review the code inside /path/to/my-app using the prompt in Mia(workspace: Prompts, folder: Code review, note: Python performance) and save the results in Mia(workspace: QA, folder: Senseibox) as "My First Code Review"

Codex should use the Mianotes MCP server to find the prompt, read it, run the review, and create a new note in the QA workspace.

You'll see something like this:

Worked for 49s
Saved to Mia in QA / Senseibox as My First Code Review.

Note ID: 89fc7784-e623-460d-b064-1a1e6daca5a5

View the result in Mianotes

Go back to Mianotes. Click the home icon, switch to the QA workspace, then open the Senseibox folder. Inside that folder, you should find:

My First Code Review

That note now contains the output from Codex. You can read it in the web app, edit it, tag it, share it, or keep it as part of the project QA history. You can also find it on disk under the QA workspace as Markdown, and add it to your git repo.

Why this workflow works well

This setup gives you a clean separation between reusable prompts and generated outputs.

Your prompts live in:

~/Desktop/projects/senseibox/prompts

Your QA results live in:

~/Desktop/projects/senseibox/qa

Both are local, plain Markdown, and versioned. They can sit inside your project structure.

That makes Mianotes useful as a shared memory layer between you and your agents. You can write the prompt once, reuse it from Codex, and keep the result where agents can find it, read it, and use it to fix issues. You also avoid losing useful agent output inside a one-off chat session. Codex can save its findings back into Mianotes, and the next agent can search or reuse that context later.

A simple pattern to repeat

Once this is working, you can create more prompt folders.

For example:

Prompts / Code review
Prompts / Security
Prompts / Release checks
Prompts / Architecture
Prompts / Documentation

And you can save agent outputs into matching QA folders:

QA / Senseibox
QA / Release reviews
QA / Security audits
QA / Architecture notes

The important part is the habit. Store the prompt once. Ask the agent to use it by name. Save the output back into Mianotes. That turns your local project folder into a working memory system for your codebase, your prompts, and your agents.

This also gives you a practical way to automate repetitive engineering tasks while keeping control of the process. Each agent starts from a clear prompt, works from the same project context, and saves the result where you can review it. That matters because agent work needs feedback loops. You need to see what the agent did, why it made those decisions, and how the output changes over time.

This is the first step towards a more autonomous team of agents, where your job is to shape the reasoning, review the decisions, and improve the evaluation process.

Thanks for reading. If you try this workflow, start small: one prompt, one agent, one saved result. Then build from there.
